### EJB(Enterprise JavaBeans)

- **자바 EE(Java Enterprise Edition)** 환경에서 서버 쪽 비즈니스 로직을 작성하고 배포하기 위해 만들어진 **컴포넌트**(재사용 가능한 독립 모듈) **기반 프레임워크**이다.
- Spring이 등장하기 전(1998년~2003년 무렵) 대규모 엔터프라이즈 애플리케이션 개발에서 사실상 표준처럼 쓰였다.
- **단점: 설정 복잡성, 개발 생산성 낮음, 벤더 종속성(이식성이 떨어짐)**

### 스프링

- Spring(2003~2004년 출시)은 이런 EJB의 복잡성과 무거움을 비판하면서 등장하게 되었다.
- 객체 지향 언어가 가진 강력한 특징을 살려내는 자바 기반 프레임워크이다.
- 좋은 객체 지향 애플리케이션을 개발할 수 있게 도와주는 프레임워크이다.

### 객체 지향 프로그래밍

- 컴퓨터 프로그램을 여러 개의 독립된 단위인 객체들의 모임으로 파악하고자 하는 것이다.
- 각각의 객체는 메시지를 주고 받고, 데이터를 처리할 수 있다.
- 유연하고 변경이 용이하기에 대규모 소프트웨어 개발에 많이 사용된다.
- 다형성을 가진다.

### SOLID : 좋은 객체 지향 설계의 5가지 원칙

- SRP: 단일 책임 원칙(single responsibility principle)
    - 한 클래스는 하나의 책임만 가져야 한다.
- OCP: 개방-폐쇄 원칙 (Open/closed principle)
    - 확장에는 열려 있으나 변경에는 닫혀 있어야 한다.
    - 클라이언트를 변경하지 않고 = **기존 코드를 수정하지 않는다(Closed)**
    - 서버의 구현 기능을 유연하게 변경할 수 있어야 한다. = **새로운 기능 추가나 동작 변경은 확장(상속, 전략 패턴, 의존성 주입 등)을 통해 수행한다(Open)**
- LSP: 리스코프 치환 원칙 (Liskov substitution principle)
    - **상속 관계**에서 하위 클래스는 상위 클래스의 계약(Contract)을 지켜야 한다.
    - 상위 타입을 사용하는 코드(클라이언트)는 **하위 타입이 들어와도 동작이 깨지면 안 된다**.
    - 단순히 컴파일만 통과하는 “형식적” 대체 가능이 아니라, **의미적(행위적) 호환성**까지 만족해야 한다.
- ISP: 인터페이스 분리 원칙 (Interface segregation principle)
    - **하나의 거대한 인터페이스보다 여러 개의 작은 인터페이스가 낫다.**
    - 불필요한 의존성 제거: **인터페이스가 너무 많은 기능을 한꺼번에 담고 있으면, 클라이언트는 쓰지 않는 메서드까지 구현하거나 의존하게 된다.**
    - **응집도 높은 인터페이스를 여러 개로 분리하면, 각 클라이언트는 필요한 기능만 사용하고 구현하면 된다.**
- DIP: 의존관계 역전 원칙 (Dependency inversion principle)
    - 추상화는 구체적인 것에 의존하면 안 되고, 구체적인 것이 추상화에 의존해야 한다.
    - **고수준 모듈(High-level Module)**: 시스템의 핵심 정책이나 비즈니스 로직을 담는 상위 계층.
    - **저수준 모듈(Low-level Module)**: 구체적 기술이나 세부 구현(데이터베이스 접근, 네트워크 처리 등)을 담당.
    - 전통적인 설계에서는 고수준 모듈이 저수준 모듈을 직접 의존 → 변경 시 상위 모듈까지 수정해야 하는 문제 발생
    - **DIP**는 이 의존 관계를 “역전”시켜, **저수준 모듈이 상위 정책의 추상화에 맞추도록 강제**해 변경 영향도를 최소화

### 객체 지향 설계와 스프링

- 다형성만으로는 OCP, DIP를 지킬 수 없다.
- 스프링은 DI 기술로 OCP와 DIP가 가능해지도록 지원한다.
    - DI = Dependency Injection : 의존관계, 의존성 주입
